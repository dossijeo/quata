# Inventario de migración multiplataforma

Actualizado durante la FASE 6. Este documento separa lógica compartible de adaptadores de plataforma; no autoriza mover varias áreas a la vez.

| Área actual | Estado | Trabajo pendiente |
| --- | --- | --- |
| `core` | Parcialmente KMP | Modelos, navegación/deep links, preferencias de sesión/tema/Touch Flow y contratos de cámara, portapapeles, compartir, preferencias, selector de archivos, permisos, ubicación, notificaciones y media se comparten; `FilePickerService` modela documentos, galería y captura. El navegador usa el selector real (`<input type=file>`) con `capture` cuando el agente lo soporta y aplica `image/*` por defecto para galería/cámara, en paridad con la petición visual Android; Android selecciona documentos con Storage Access Framework y galería con `GetContent`, mientras captura nativa sigue explícitamente fuera de este contrato. Android ofrece adaptadores reales de portapapeles, preferencias, compartir, ubicación y estado de permisos (el prompt se inyecta desde el host con launcher), mientras Media3, cámara nativa, notificaciones, almacenamiento y bridges de sistema siguen como adaptadores; iOS compone preferencias (`NSUserDefaults`) y portapapeles (`UIPasteboard`, ambos inyectables), compartir mediante `UIActivityViewController`, selector mediante `UIDocumentPickerViewController`, autorización de notificaciones (`UNUserNotificationCenter`, capacidad disponible sin APNs) y Core Location reales detrás de `IosPlatformServices` cuando el launcher inyecta sus hosts UIKit/Core Location. ## Orden de ejecución

1. Auth: crear módulo KMP y extraer contrato, estados, eventos y ViewModels; Google Sign-In queda como adaptador.
2. Communities y Notifications: un módulo por feature, sin mover sus repositorios Android.
3. About/release history y External share: extraer primero modelos y lógica sin intents.
4. SOS: separar coordinador común de ubicación, permisos, notificaciones e intents Android.
5. Traductor Fang: migrar el renderer Compose del overlay; conservar detector, caché y captura como adaptadores de plataforma.
6. Quata Touch Flow: extraer la preferencia por usuario mediante el contrato multiplataforma de preferencias.
7. Fondos procedurales de chat: extraer semilla, selección de paleta, clave y contrato de renderizado; mantener EGL/GLES, WebP y ficheros como adaptadores hasta contar con implementaciones JS/iOS.
8. Rich text, documento y multimedia: definir contratos de plataforma y mover sólo la lógica/UI que compile en `commonMain`.
